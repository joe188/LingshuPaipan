# 开发工作规范

## 调试与问题处理原则

**核心原则：遇到困难时，坚持方案、先寻解法、及时沟通**

当开发过程中遇到技术困难（如依赖冲突、构建失败、API不兼容等）时，必须遵守以下流程：

### 1. 坚持当前方案
- **不擅自更换方案**：遇到困难时不得随意切换到替代方案
- **不降级依赖**：未经沟通，不得擅自降级依赖版本
- **保持配置稳定**：避免反复修改配置文件导致状态混乱

### 2. 主动搜索解决方案
- **查找相关 Skill**：优先搜索 OpenClaw 生态中是否有对应的技术 Skill
- **网上搜索案例**：搜索相似问题的解决方案（GitHub Issues、Stack Overflow、官方文档）
- **查阅依赖文档**：确认依赖版本兼容性，查找版本适配关系

### 3. 及时沟通汇报
- **说明问题**：清晰描述遇到的问题、错误信息、已尝试的方法
- **请示决策**：在尝试多种方案无果后，及时向团队/用户请示下一步
- **记录过程**：将问题排查过程和解决方案写入 memory 文件

### 4. 规范执行流程
```
遇到问题 → 坚持原方案 → 搜索 Skill → 网上查案例 → 版本适配检查
    ↓
仍无法解决 → 及时沟通 → 获取明确指示 → 执行决策
    ↓
问题解决 → 记录教训 → 更新文档
```

### 5. 网络与镜像优化
- **国内环境优先使用国内镜像源**（如阿里云 maven.aliyun.com，用于 Gradle/Maven 依赖下载）
- 对于 Gradle 插件依赖，可通过 `~/.gradle/init.d` 统一配置 mirror，避免网络 TLS/超时问题
- 修改仓库配置前需在 `DEVELOPMENT_STANDARDS.md` 中记录使用的镜像地址和生效范围

## 违反规范的后果
擅自更换方案、降级依赖可能导致：
- 引入新的兼容性问题
- 与现有架构不匹配
- 浪费开发时间走弯路
- 破坏已验证的配置
